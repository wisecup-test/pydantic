<rule_activation id="6c7d2b3e-b228-43c5-b293-31d8a5e3ebf8" title="Standardize Testing Fixtures for Public API Validation: Fixtures Organized Functional" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files matching public API modules, versioned implementations, and internal implementation modules that directly support public APIs.
</rule_activation>

### Rules

- **R-FIXTURE-001** SHOULD: Fixtures SHOULD be organized by functional domain (networks, schema generation, model construction) to improve maintainability.

### Verify

```bash
# Count test functions across the codebase
grep -r 'def test_' tests/ | wc -l

# Find pytest fixture definitions
find tests/ -name 'conftest.py' -o -name 'fixtures.py' | xargs grep -l '@pytest.fixture'

# Count tests related to networks, schema, and model domains
pytest --collect-only -q | grep -E '(networks|schema|model)' | wc -l
```

**Accept when:**
- All public API modules have corresponding test files with fixture-based validation
- Test coverage for public API modules exceeds 80% with fixtures covering both happy path and edge cases
- Versioned API implementations have separate fixture sets that validate version-specific behavior
- Fixtures are organized in functional domain directories (e.g., fixtures/networks/, fixtures/schema/, fixtures/models/)
- Fixture documentation exists explaining the purpose and expected behavior of each fixture set

<enforcement>
Claude Code MUST NOT skip or defer verification. Pull requests adding or modifying public APIs without corresponding fixtures are blocked from merge. Coverage drops below 80% threshold trigger build failures and require remediation before release.
</enforcement>
