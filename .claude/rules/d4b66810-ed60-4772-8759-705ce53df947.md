<rule_activation id="d4b66810-ed60-4772-8759-705ce53df947" title="Adopt Environment Variable-Based Secret Management for External API Integration: Environment Variable Names" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files matching `**/*`. Environment variable-based secret management is required for all external API integrations requiring authentication credentials.
</rule_activation>

### Rules

- **R-SEC-001** MUST: Environment variable names for API credentials MUST follow a consistent naming convention (e.g., SERVICE_NAME_API_KEY, SERVICE_NAME_TOKEN)

### Verify

```bash
# Check for hardcoded API keys in Python files
grep -r "api[_-]key\s*=\s*['\"]" --include="*.py" --exclude-dir=tests --exclude-dir=.venv | grep -v "os.environ" | grep -v "getenv"

# Check for hardcoded tokens in Python files
grep -r "token\s*=\s*['\"]" --include="*.py" --exclude-dir=tests --exclude-dir=.venv | grep -v "os.environ" | grep -v "getenv"

# Check for .env files not properly excluded from version control
find . -name ".env" -not -path "*/node_modules/*" -not -path "*/.venv/*" | xargs git check-ignore -v
```

**Accept when:**
- No hardcoded API credentials are found in source code (excluding test mocks)
- All .env files containing actual secrets are properly excluded from version control
- All external API integrations retrieve credentials from environment variables at runtime
- Documentation clearly specifies required environment variables for each integration
- Environment variable names follow the SERVICE_NAME_API_KEY or SERVICE_NAME_TOKEN convention

<enforcement>
Claude Code MUST NOT skip or defer verification. All pull requests and code changes must pass the verification commands above before acceptance. Violations require immediate credential rotation and security team notification.
</enforcement>
