<rule_activation id="f31763d1-95eb-40ac-bdcc-a490b0862315" title="Adopt Environment Variable-Based Secret Management for External API Integration: Documentation Clearly Specify" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files matching `**/*`. Environment variable-based secret management is required for all external API integrations requiring authentication credentials across development, staging, production environments, CI/CD pipelines, and documentation build processes.
</rule_activation>

### Rules

- **R-SEC-001** SHOULD: Documentation SHOULD clearly specify which environment variables are required for each external API integration.

### Verify

```bash
# Check for hardcoded API credentials in Python files
grep -r "api[_-]key\s*=\s*['\"]" --include="*.py" --exclude-dir=tests --exclude-dir=.venv | grep -v "os.environ" | grep -v "getenv"

# Check for hardcoded tokens in Python files
grep -r "token\s*=\s*['\"]" --include="*.py" --exclude-dir=tests --exclude-dir=.venv | grep -v "os.environ" | grep -v "getenv"

# Check for .env files not properly excluded from version control
find . -name ".env" -not -path "*/node_modules/*" -not -path "*/.venv/*" | xargs git check-ignore -v
```

**Accept when:**
- No hardcoded API credentials are found in source code (excluding test mocks)
- All .env files containing actual secrets are properly excluded from version control
- All external API integrations retrieve credentials from environment variables at runtime
- Documentation clearly specifies required environment variables for each integration

<enforcement>
Claude Code MUST NOT skip or defer verification. All external API integrations MUST use environment variables for credentials. Hardcoded secrets are security violations and must be rejected.
</enforcement>
