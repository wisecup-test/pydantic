<rule_activation id="db85423a-8fd3-4e18-9877-f170255cdaa2" title="Enforce Type Safety and Validation in Public API Contracts: Internal Between Trusted" applies_to="**/*">
These rules are ALWAYS ACTIVE for all public API contracts, external integrations, and type validation implementations. All code that exposes public APIs or validates external data MUST comply with these rules.
</rule_activation>

### Rules

- **R-API-001** MUST: All REST API endpoints exposed to external consumers require Pydantic models with 100% field coverage for type validation and runtime contract enforcement.
- **R-API-002** MUST: All GraphQL schemas and resolvers accessible outside the organization MUST include comprehensive type definitions and validation.
- **R-API-003** MUST: SDK and library public interfaces MUST define explicit type contracts using Pydantic BaseModel or equivalent runtime validation.
- **R-API-004** MUST: Webhook handlers receiving data from third-party services MUST validate all incoming data against defined Pydantic schemas before processing.
- **R-API-005** MUST: Data ingestion pipelines processing external data sources MUST implement runtime validation at the API boundary.
- **R-API-006** MUST: Configuration files and environment variables affecting public API behavior MUST be validated against defined schemas.
- **R-TYPE-001** MUST: Configure mypy with strict mode in CI/CD pipeline for all public API modules.
- **R-TYPE-002** MUST: All public API modules MUST pass mypy strict mode type checking with zero errors.
- **R-TYPE-003** SHOULD: Use Pydantic V2 for new implementations to benefit from performance improvements via Rust core (pydantic-core).
- **R-VAL-001** MUST: Implement comprehensive error handling that translates validation errors into appropriate HTTP status codes (400 Bad Request) with detailed error messages.
- **R-VAL-002** SHOULD: Create shared base models and validators for common patterns (pagination, timestamps, IDs) to ensure consistency across APIs.
- **R-VAL-003** SHOULD: Document all public models with docstrings and examples that are automatically included in API documentation.
- **R-PERF-001** MAY: Internal APIs between trusted services MAY use relaxed validation if performance requirements justify it and risks are documented.
- **R-EXC-001** EXCEPTION: Performance-critical internal APIs where validation overhead exceeds 10% of request latency and all consumers are trusted internal services MAY defer strict validation with documented risk assessment.
- **R-EXC-002** EXCEPTION: Gradual migration of legacy APIs where immediate strict validation would break existing consumers MAY use phased rollout with deprecation warnings.

### Verify

```bash
# Count Pydantic models in codebase
grep -r "class.*BaseModel" --include="*.py" | wc -l

# Verify static type checking with mypy strict mode
mypy --strict --config-file mypy.ini . 2>&1 | grep -E "(error|Success)"

# Run type validation tests
pytest tests/test_typing.py tests/test_errors.py -v

# Count custom validators
grep -r "@validator\|@field_validator" --include="*.py" | wc -l
```

**Accept when:**
- All public API endpoints have corresponding Pydantic models with 100% field coverage
- Mypy strict mode passes with zero errors for all public API modules
- Test coverage for validation logic exceeds 90% with tests for invalid inputs, edge cases, and error messages
- CI/CD pipeline includes automated validation checks that fail builds on type errors or missing validation
- All REST, GraphQL, SDK, webhook, and data ingestion APIs include runtime validation at boundaries
- Validation errors are translated to appropriate HTTP status codes with detailed error messages

</enforcement>
