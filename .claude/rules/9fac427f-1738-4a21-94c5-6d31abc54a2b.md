<rule_activation id="9fac427f-1738-4a21-94c5-6d31abc54a2b" title="Adopt Environment Variable-Based Secret Management for External API Integration: Projects Use Additional" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files matching `**/*`. External API integrations requiring authentication credentials must use environment variable-based secret management to prevent exposure in version control systems and maintain security best practices.
</rule_activation>

### Rules

- **R-SEC-001** MUST: Never hardcode API keys, tokens, OAuth credentials, or other authentication secrets in source code.
- **R-SEC-002** MUST: Retrieve all external API credentials from environment variables at runtime using `os.environ.get()` or `os.getenv()` with appropriate default handling and validation.
- **R-SEC-003** MUST: Add `.env` files containing actual secrets to `.gitignore` to prevent accidental commits of credentials.
- **R-SEC-004** MUST: Create a `.env.example` file documenting all required environment variables with placeholder values (e.g., `GITHUB_API_TOKEN=your_token_here`).
- **R-SEC-005** MUST: Implement startup validation that checks for required environment variables and fails fast with clear error messages.
- **R-SEC-006** MUST: Use consistent naming conventions for environment variables: `SERVICE_NAME_API_KEY`, `SERVICE_NAME_TOKEN`, `SERVICE_NAME_SECRET`.
- **R-SEC-007** SHOULD: Document environment variable requirements in README.md and deployment documentation.
- **R-SEC-008** SHOULD: Implement logging filters to redact sensitive environment variables and use structured logging with secret masking.
- **R-SEC-009** MAY: Use additional secret management tools (e.g., HashiCorp Vault, AWS Secrets Manager) that inject secrets as environment variables for large-scale production deployments.
- **R-SEC-010** MAY: Unit tests use hardcoded mock credentials when testing against mock API servers with no external connectivity (EXC-001).

### Verify

```bash
# Check for hardcoded API credentials in Python files
grep -r "api[_-]key\s*=\s*['\"]" --include="*.py" --exclude-dir=tests --exclude-dir=.venv | grep -v "os.environ" | grep -v "getenv"

# Check for hardcoded tokens in Python files
grep -r "token\s*=\s*['\"]" --include="*.py" --exclude-dir=tests --exclude-dir=.venv | grep -v "os.environ" | grep -v "getenv"

# Verify .env files are properly excluded from version control
find . -name ".env" -not -path "*/node_modules/*" -not -path "*/.venv/*" | xargs git check-ignore -v
```

**Accept when:**
- No hardcoded API credentials are found in source code (excluding test mocks)
- All `.env` files containing actual secrets are properly excluded from version control
- All external API integrations retrieve credentials from environment variables at runtime
- Documentation clearly specifies required environment variables for each integration
- A `.env.example` file exists documenting all required environment variables with placeholder values
- Startup validation checks for required environment variables and fails with clear error messages

<enforcement>
Claude Code MUST NOT skip or defer verification. All pull requests containing hardcoded credentials must be rejected. Exposed credentials must be revoked and regenerated before code can be merged. Security team must be notified of any credential exposure incidents. Violations require incident post-mortem for production exposures.
</enforcement>
