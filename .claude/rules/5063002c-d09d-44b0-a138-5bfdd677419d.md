These rules are ALWAYS ACTIVE for all files in the public API surface, versioned implementations, and internal modules that directly support public APIs.
</rule_activation>

### Rules

- **R-5063002c-001** MUST: All public API modules MUST include testing fixtures that validate expected behavior against documented contracts.

### Scope

In scope:
- All modules in public API surface (networks, validation, schema generation)
- Versioned API implementations (v1, v2, etc.)
- Internal implementation modules that directly support public APIs (_internal/_decorators, _internal/_model_construction, _internal/_generate_schema)
- API contract validation and backward compatibility testing

Out of scope:
- Internal utility functions not exposed through public APIs
- Performance benchmarking tests (separate from functional validation)
- Integration tests that span multiple services
- Documentation examples (unless they serve as executable fixtures)

### Verify

```bash
# Count test functions in test suite
grep -r 'def test_' tests/ | wc -l

# Find pytest fixture definitions
find tests/ -name 'conftest.py' -o -name 'fixtures.py' | xargs grep -l '@pytest.fixture'

# Count tests related to public API modules
pytest --collect-only -q | grep -E '(networks|schema|model)' | wc -l
```

**Accept when:**
- All public API modules have corresponding test files with fixture-based validation
- Test coverage for public API modules exceeds 80% with fixtures covering both happy path and edge cases
- Versioned API implementations have separate fixture sets that validate version-specific behavior

### Implementation Guidelines

- Organize fixtures by functional domain (e.g., fixtures/networks/, fixtures/schema/, fixtures/models/)
- Use pytest fixtures or similar framework features to enable fixture reuse and composition
- Document the purpose and expected behavior of each fixture set
- Implement fixture generators for parameterized testing across multiple input variations
- Consider snapshot testing for complex API responses

Claude Code MUST NOT skip or defer verification. All public API modules MUST include testing fixtures. Pull requests adding or modifying public APIs without corresponding fixtures are blocked from merge. Coverage drops below 80% threshold trigger build failures. Quarterly test suite health reviews identify and prioritize fixture gaps.
</enforcement>
