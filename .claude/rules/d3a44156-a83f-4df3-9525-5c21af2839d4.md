<rule_activation id="d3a44156-a83f-4df3-9525-5c21af2839d4" title="Establish Pydantic as Standard Data Validation and Configuration Framework: Datetime Parsing Validation" applies_to="**/*.py">
These rules are ALWAYS ACTIVE for all Python codebases utilizing data validation, configuration management, or API contract enforcement.
</rule_activation>

### Rules

- **R-PYDANTIC-001** MUST: Datetime parsing and validation MUST utilize Pydantic's built-in datetime validators rather than custom parsing logic.
- **R-PYDANTIC-002** MUST: All REST API endpoints receiving or returning structured data MUST use Pydantic models for request/response types.
- **R-PYDANTIC-003** MUST: Configuration file parsing (YAML, JSON, TOML) MUST use Pydantic BaseModel subclasses.
- **R-PYDANTIC-004** MUST: External client SDK interfaces MUST define data transfer objects (DTOs) as Pydantic models.
- **R-PYDANTIC-005** MUST: CLI argument parsing and validation MUST use Pydantic models.
- **R-PYDANTIC-006** SHOULD: Use Pydantic's ConfigDict to set validation behavior (strict mode, arbitrary_types_allowed, etc.) consistently across the codebase.
- **R-PYDANTIC-007** SHOULD: Leverage Field() for adding constraints (min_length, max_length, regex patterns) and metadata (description, examples) to improve API documentation.
- **R-PYDANTIC-008** SHOULD: For datetime handling, configure timezone handling explicitly in model config.
- **R-PYDANTIC-009** MAY: Internal function parameter validation within a single module may use type hints only without Pydantic validation.
- **R-PYDANTIC-010** MAY: Simple scripts with no external interfaces may use alternative validation approaches.

### Exceptions

- **EXC-001**: Performance profiling demonstrates Pydantic validation causes >10% latency increase in critical path (requires documentation and tech lead approval).
- **EXC-002**: Third-party library integration requires incompatible validation framework (requires documentation and tech lead approval).

### Verify

```bash
# Count Pydantic imports
grep -r "from pydantic import" --include="*.py" | wc -l

# Count BaseModel subclasses (excluding tests)
grep -r "class.*BaseModel" --include="*.py" | grep -v "test" | wc -l

# Check Pydantic version
python -c "import pydantic; print(f'Pydantic version: {pydantic.VERSION}')"

# Count field and model validators
grep -r "@field_validator\|@model_validator" --include="*.py" | wc -l

# Verify no manual datetime.strptime in API handlers
grep -r "datetime.strptime" --include="*.py" | grep -v "test" | grep -v "#"
```

**Accept when:**
- All API endpoint handlers use Pydantic models for request/response types (verified by grep showing BaseModel usage in route definitions)
- Configuration loading modules import from pydantic and define config classes as BaseModel subclasses
- Datetime parsing uses Pydantic validators rather than manual datetime.strptime calls
- Code review checklist includes verification that new external interfaces use Pydantic validation
- New API endpoints include Pydantic model definitions in CI pipeline checks
