<rule_activation id="beef152b-70ce-488b-935a-9bd630dcd7e7" title="Adopt Runtime Configuration Sources for Plugin and Extension Systems: Configuration Loaders Not" applies_to="**/*">
These rules are ALWAYS ACTIVE for all public/external API implementations that support plugin architectures, dynamic configuration loading, or extensibility mechanisms requiring runtime configuration sources.
</rule_activation>

### Rules

- **R-RUNTIME-CONFIG-001** MUST_NOT: Configuration loaders MUST NOT expose sensitive credentials or secrets in error messages or logs

### Verify

```bash
# Detect runtime configuration source patterns
grep -r 'runtime.*config.*source\|config.*loader\|load.*config' --include='*.py' | grep -v test | wc -l

# Check for configuration imports in plugin files
python -c "import ast; import sys; tree=ast.parse(open(sys.argv[1]).read()); print(any(isinstance(n, ast.Import) and any('config' in a.name for a in n.names) for n in ast.walk(tree)))" */plugin*.py

# Find configuration loading code in plugin/loader/schema files
find . -name '*.py' -exec grep -l 'environ\|getenv\|config.*load\|load.*config' {} \; | grep -E '(plugin|loader|schema)' | head -5
```

**Accept when:**
- At least one configuration loader implementation is detected in plugin or extension system files
- Configuration loading code includes error handling for missing or invalid configuration sources
- API documentation includes a configuration section describing supported sources and precedence rules
- Test suite includes tests for configuration loading from multiple sources (environment variables, files, defaults)
- Configuration loading code does not expose secrets in error messages or logs

<enforcement>
Claude Code MUST verify all configuration loaders comply with R-RUNTIME-CONFIG-001 and MUST NOT skip verification for plugin systems, extension mechanisms, or runtime configuration implementations.
</enforcement>
