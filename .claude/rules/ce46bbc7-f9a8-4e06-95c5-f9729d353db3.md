<rule_activation id="ce46bbc7-f9a8-4e06-95c5-f9729d353db3" title="Standardize Testing Fixtures for Public API Validation: Internal Implementation Modules" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files matching public API modules, versioned implementations, and internal implementation modules that support public APIs.
</rule_activation>

### Rules

- **R-FIXTURE-001** SHOULD: Internal implementation modules that support public APIs SHOULD use the same fixture-based testing approach to ensure consistency with public API surface testing.

### Verify

```bash
# Count test functions across the codebase
grep -r 'def test_' tests/ | wc -l

# Find pytest fixture definitions
find tests/ -name 'conftest.py' -o -name 'fixtures.py' | xargs grep -l '@pytest.fixture'

# Count tests related to networks, schema, and model modules
pytest --collect-only -q | grep -E '(networks|schema|model)' | wc -l
```

**Accept when:**
- All public API modules have corresponding test files with fixture-based validation
- Test coverage for public API modules exceeds 80% with fixtures covering both happy path and edge cases
- Versioned API implementations have separate fixture sets that validate version-specific behavior
- Internal implementation modules (_internal/_decorators, _internal/_model_construction, _internal/_generate_schema) use the same fixture patterns as public modules

<enforcement>
Claude Code MUST NOT skip or defer verification. All public API changes require corresponding fixture-based tests. Pull requests adding or modifying public APIs without fixtures are blocked from merge. Coverage drops below 80% threshold trigger build failures.
</enforcement>
