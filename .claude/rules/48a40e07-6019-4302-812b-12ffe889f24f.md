<rule_activation id="48a40e07-6019-4302-812b-12ffe889f24f" title="Adopt Runtime Configuration Sources for Plugin and Extension Systems: Public Plugin Extension" applies_to="**/*">
These rules are ALWAYS ACTIVE for all public-facing APIs that implement plugin architectures, external APIs with extensibility mechanisms requiring runtime configuration, documentation generation systems with configurable plugins, schema validation systems with dynamic configuration requirements, and third-party integration points requiring environment-specific settings.
</rule_activation>

### Rules

- **R-RUNTIME-CONFIG-001** MUST: Public APIs with plugin or extension systems MUST support runtime configuration sources that can be loaded without modifying source code.

### Verify

```bash
# Check for runtime configuration and config loading patterns
grep -r 'runtime.*config.*source\|config.*loader\|load.*config' --include='*.py' | grep -v test | wc -l

# Verify configuration imports in plugin files
python -c "import ast; import sys; tree=ast.parse(open(sys.argv[1]).read()); print(any(isinstance(n, ast.Import) and any('config' in a.name for a in n.names) for n in ast.walk(tree)))" */plugin*.py

# Find configuration loading code in plugin/loader/schema files
find . -name '*.py' -exec grep -l 'environ\|getenv\|config.*load\|load.*config' {} \; | grep -E '(plugin|loader|schema)' | head -5
```

**Accept when:**
- At least one configuration loader implementation is detected in plugin or extension system files
- Configuration loading code includes error handling for missing or invalid configuration sources
- API documentation includes a configuration section describing supported sources and precedence rules
- Test suite includes tests for configuration loading from multiple sources (environment variables, files, defaults)

<enforcement>
Claude Code MUST NOT skip or defer verification. Configuration loading patterns must be validated before approving plugin or extension system implementations.
</enforcement>
